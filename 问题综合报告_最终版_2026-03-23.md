# KidsPhoneGuard 项目问题综合报告（无障碍掉权 + 限时拦截）

## 报告目的

本报告用于向外部专家快速说明当前项目状态、已确认问题、证据链、剩余风险与建议排查路径，目标是尽量减少二次沟通成本。

报告时间：2026-03-23
项目路径：`V:\phone ctrl`
目标设备：华为系 Android 设备（USB 调试已接入）

---

## 1. 项目定位与核心工作机制（简述）

KidsPhoneGuard 是"儿童终端应用访问控制"项目，核心依赖：

1. **无障碍服务**：识别应用切换并执行拦截动作（遮罩、返回/回桌面、兜底压制）。
2. **使用统计权限**：统计应用使用时长，驱动"限时规则"触发。
3. **前台守护服务**：持续保活、权限状态观察、异常状态提示。

关键模块：

- `GuardAccessibilityService`（核心拦截执行）
- `UsageTrackingManager`（时长统计与限时触发）
- `LockDecisionEngine`（统一策略决策）
- `GuardForegroundService`（保活 + 权限/健康监测 + 取证）

---

## 2. 当前主要异常现象（用户实测）

1. 某些受限应用在应拦截时段或超时后，仍可继续使用。
2. 再次打开家长端后，拦截能力可能恢复。
3. 出现"无障碍权限掉了"的情况，表现为系统层开关关闭或服务不在启用列表。
4. 用户体感上"像后台被杀/功能不生效"。

---

## 3. 已确认事实（证据充分）

### 3.1 无障碍掉权是"系统设置层关闭"，不是仅服务进程挂掉

实测命令结果出现：

- `settings secure accessibility_enabled = 0`
- `settings secure enabled_accessibility_services = null`

并且 `dumpsys accessibility` 显示：

- `Enabled services:{}`
- `Bound services:{}`
- 但 `com.kidsphoneguard/.service.GuardAccessibilityService` 仍在"已安装无障碍服务列表"中

结论：当前问题属于"系统未启用该服务"，不是 APK 中服务类消失或仅短时线程卡死。

### 3.2 掉权期间前台守护仍在运行，但守护状态持续 degraded

健康快照长期出现：

- `ae=false`
- `ar=false`
- `aMissing=true`
- `degraded=true`

说明：前台服务存活、循环在跑，但核心无障碍链路不可用。

### 3.3 早期限时链路存在误判风险（已实施改造）

历史问题点：

- 仅依赖 `UsageStats` 最近记录推断前台，可能被 launcher/系统应用扰动。
- 时间窗判定边界与异常容错不足，存在漏拦截窗口。

已改造：

- 前台识别：`UsageEvents` 优先，`UsageStats` 兜底。
- 限时触发：统一走 `LockDecisionEngine`。
- 时间窗判定：修复边界时刻与异常窗口容错。

## 4. 已完成改造清单（含影响范围）

### 4.1 限时链路健壮性改造（业务侧）

1. **前台识别增强**（降低错误归因到 Launcher 风险）
2. **时长累计上限钳制**（避免轮询抖动导致异常突增）
3. **统一决策触发**（LIMIT 不再分叉判定）
4. **时间窗边界与容错修复**
5. **无障碍决策引擎延迟初始化兜底**

### 4.2 掉权实时取证链路增强（非业务逻辑改动）

1. 新增 `accessibility_forensics` 取证日志（状态切换、设置变更、生命周期关键点触发）。
2. 新增无障碍服务生命周期信号共享（最近事件/生命周期信号）。
3. 强制快照机制（避免状态未变时关键日志被去重漏掉）。

## 5. 问题分层（已修正）

### 5.1 明确问题（可以下结论）

1. **无障碍掉权真实存在且会直接导致核心能力失效。**
2. **掉权发生时通常不是进程单点崩溃，而是系统层未启用无障碍。**
3. **限时链路过去存在健壮性不足，已做结构性补强。**
4. **项目已具备掉权现场取证能力，可抓取上下文证据链。**
5. **华为/荣耀设备的 ROM 策略会对无障碍服务产生额外影响。**

### 5.2 已排除/澄清的问题

1. **轮询间隔 3 秒导致漏记的风险**：实际影响极低，漏记仅影响排查，不影响业务逻辑正确性。
2. **LockDecisionEngine 单例初始化时序风险**：经核验，`getInstance()` 为同步方法，在 Application.onCreate 之后调用，数据库已就绪，不存在此问题。
3. **屏幕关闭时长统计停止**：这是正确的设计逻辑，不是 bug。屏幕关闭时不计时符合业务需求。

## 6. 待修复问题清单

### 🔴 P0（最高优先级）

#### 6.1 无障碍掉权后自动恢复引导链路不足

**问题描述**：
当前代码对无障碍掉权的处理以取证和状态告警为主，恢复动作不足。一旦系统在设置层关闭无障碍，核心拦截能力立即失效，恢复依赖用户再次进入设置页面手动开启。

**当前行为**：
- `GuardForegroundService` 中的 `accessibilitySettingsObserver` 仅观察和记录设置变化
- 检测到 `degraded` 状态后主要更新通知栏和取证日志

**期望行为**：
- 当 `accessibility_enabled=0` 或服务不在启用列表时，自动触发“恢复引导路径”（跳转无障碍详情页）
- 当设置层仍为启用但服务运行态异常时，优先触发取证 + 引导 + 守护服务自检重试

**修复建议**：
在 `GuardForegroundService` 中增加自动恢复引导检测：
```kotlin
private val accessibilityRecoveryRunnable = object : Runnable {
    override fun run() {
        val isEnabled = PermissionManager.isAccessibilityServiceEnabled(this@GuardForegroundService)
        val isRunning = GuardAccessibilityService.isServiceRunning()

        if (!isEnabled) {
            Log.w(TAG, "Accessibility disabled in settings, trigger guide")
            emitAccessibilityForensics("auto_recovery_guide_disabled")
            PermissionManager.requestAccessibilityPermission(this@GuardForegroundService)
        } else if (!isRunning) {
            Log.w(TAG, "Accessibility enabled but runtime not healthy, trigger guide")
            emitAccessibilityForensics("auto_recovery_guide_stale")
            PermissionManager.requestAccessibilityPermission(this@GuardForegroundService)
        }

        handler.postDelayed(this, 5000L)
    }
}
```
说明：无障碍服务受系统设置绑定约束，不应将 `startService(GuardAccessibilityService)` 作为“自动重连”主方案。

### 🟡 P1（高优先级）

#### 6.2 华为/荣耀设备 ROM 策略适配缺失

**问题描述**：
华为/荣耀设备有"辅助功能治理"、"应用启动管理"、"省电策略"等系统级机制，可能在用户不知情的情况下自动关闭无障碍服务或杀死后台进程。当前项目已做部分厂商识别与拦截策略适配，但缺专项引导与设置路径闭环。

**当前行为**：
- 已在拦截链路中识别华为/荣耀设备并采用更激进导航压制策略
- 权限页仅提供通用电池优化入口，缺少华为/荣耀专项教育提示与操作指引

**期望行为**：
- 在检测到华为设备时，主动引导用户关闭"应用启动管理"的自动管理
- 引导用户在电池优化中将应用设为"不允许"
- 提供更明确的教育性提示，告知用户华为设备需要手动保护的路径

**修复建议**：
在 `GuardForegroundService` 或 `MainActivity` 中增加华为设备专项检测：
```kotlin
private fun checkHuaweiDeviceProtection(): Boolean {
    val powerManager = getSystemService(Context.POWER_SERVICE) as PowerManager
    val isIgnoringBattery = powerManager.isIgnoringBatteryOptimizations(packageName)

    // 检测应用是否在华为的"应用启动管理"中被禁止
    // 可以通过尝试启动一个 Activity 并检查结果来判断

    return isIgnoringBattery
}
```

### 🟠 P2（中优先级）

#### 6.3 取证链路可进一步增强

**问题描述**：
当前取证日志已较为完善，但极端情况下（如系统日志被覆盖）可能丢失关键证据。

**当前已实现**：
- `accessibility_forensics` 综合日志
- `health_snapshot` 健康状态快照
- `accessibility_settings_changed` 设置变更记录
- `degraded_state_changed` 状态切换触发

**可增强方向**：
1. 增加掉权时立即输出完整进程树信息
2. 增加电池/省电策略状态快照
3. 将关键取证数据同时写入本地文件（双保险）

**修复建议**：
```kotlin
private fun emitProcessTreeForensics() {
    try {
        val activityManager = getSystemService(Context.ACTIVITY_SERVICE) as ActivityManager
        val processInfo = StringBuilder()
        activityManager.runningAppProcesses?.forEach { proc ->
            processInfo.append("${proc.processName}:${proc.importance}|")
        }
        Log.e(TAG, "process_tree: $processInfo")
    } catch (e: Exception) {
        Log.e(TAG, "process_tree_forensics failed: ${e.message}")
    }
}
```

## 9. 外部专家可直接使用的排查入口

### 9.1 现场抓取命令（建议直接复制执行）

```powershell
adb logcat -c
adb logcat -v time -s GuardForegroundService:W GuardAccessibilityService:D UsageTrackingManager:D ActivityManager:I AndroidRuntime:E
```

### 9.2 关键判读字段

- `accessibility_settings_changed`：系统无障碍开关与服务列表快照
- `accessibility_forensics`：取证汇总（top/procImp/aHbAge/signal/ignoreBattery）
- `degraded_state_changed`：状态切换触发点
- `health_snapshot`：持续健康指标

### 9.3 关键判读逻辑

1. 先看 `ae`（系统无障碍总开关）是否从 1 变 0。
2. 再看 `enabledServices` 是否移除 `com.kidsphoneguard/.service.GuardAccessibilityService`。
3. 对齐同时间窗 `top`、`procImp`、`signal` 判断是：
   - 设置页操作路径
   - 系统策略触发路径
   - 进程级异常路径
