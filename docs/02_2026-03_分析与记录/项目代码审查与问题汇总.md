## 文档说明
本文档汇总了对 KidsPhoneGuard 项目进行全面代码审查后发现的问题，包括安全风险、性能问题、架构问题和代码质量问题。

**创建日期**: 2026-03-15
**审查范围**: 全项目代码扫描

## 一、安全问题（高优先级）

### 1.1 密码明文存储

**位置**: `PasswordManager.kt`

**问题描述**:
- 密码以明文形式存储在 SharedPreferences 中
- 存在默认弱口令 `1234`

**安全风险**:
- 如果手机被 root，任何应用都可以读取 SharedPreferences
- 默认弱口令容易被猜测或暴力破解
- 日志中可能泄露密码

**代码位置**:
```kotlin
// 可能的代码模式（需验证）
private const val DEFAULT_PASSWORD = "1234"
fun savePassword(password: String) {
    preferences.edit().putString(KEY_PASSWORD, password).apply()
}
```

**建议修复**:
```kotlin
import javax.crypto.SecretKeyFactory
import javax.crypto.spec.PBEKeySpec

// 使用加盐哈希
fun hashPassword(password: String, salt: ByteArray): String {
    val spec = PBEKeySpec(password.toCharArray(), salt, 10000, 256)
    val factory = SecretKeyFactory.getInstance("PBKDF2WithHmacSHA256")
    val hash = factory.generateSecret(spec).encoded
    return Base64.encodeToString(hash, Base64.NO_WRAP)
}

// 移除默认密码，强制首次设置
```

### 1.2 白名单包名匹配漏洞

**位置**: `WhitelistManager.kt` 第 265 行

**问题描述**:
使用 `contains()` 进行包名匹配，存在包名伪装绕过风险。

**代码**:
```kotlin
fun isInWhitelist(packageName: String): Boolean {
    return SYSTEM_WHITELIST.any { packageName.contains(it, ignoreCase = true) }
}
```

**攻击示例**:
```
白名单: com.miui.home
恶意包名: com.miui.home.evil.stealer
结果: ✅ 被放过（危险！）
```

**建议修复**:
```kotlin
fun isInWhitelist(packageName: String): Boolean {
    // 精确匹配系统关键应用
    val exactMatch = SYSTEM_WHITELIST.contains(packageName)
    if (exactMatch) return true

    // 仅对明确的系统包前缀进行前缀匹配
    val systemPrefixes = setOf(
        "com.android.",
        "com.miui.",
        "com.huawei.",
        "com.samsung.android."
    )
    return systemPrefixes.any { packageName.startsWith(it) && SYSTEM_WHITELIST.any {
        it.startsWith(packageName) || packageName.startsWith("$it.")
    }}
}
```

### 1.3 广播无权限保护

**位置**: `BroadcastPermissionHelper.kt`

**问题描述**:
自定义广播 `ACTION_BLOCK_APP` 和 `ACTION_KILL_APP` 没有权限保护，任何应用都可以发送。

**代码**:
```kotlin
const val ACTION_BLOCK_APP = "com.kidsphoneguard.action.BLOCK_APP"
const val ACTION_KILL_APP = "com.kidsphoneguard.action.KILL_APP"

fun sendInternalBroadcast(context: Context, action: String, extras: Map<String, String> = emptyMap()): Boolean {
    val intent = Intent(action)
    extras.forEach { (key, value) -> intent.putExtra(key, value) }
    context.sendBroadcast(intent)  // ⚠️ 无权限保护
    return true
}
```

**安全风险**:
- 恶意应用可以发送 `ACTION_KILL_APP` 来杀死任意应用
- 可以触发虚假的 `ACTION_BLOCK_APP` 来干扰正常功能

**建议修复**:

方案1: 使用保护性广播
```kotlin
// Android 12+ 使用 RECEIVER_NOT_EXPORTED
if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.TIRAMISU) {
    context.registerReceiver(receiver, filter, Context.RECEIVER_NOT_EXPORTED)
}
```

方案2: 添加自定义权限
```xml
<!-- AndroidManifest.xml -->
<permission
    android:name="com.kidsphoneguard.permission.RECEIVE_BLOCK_BROADCAST"
    android:protectionLevel="signature" />

<receiver
    android:permission="com.kidsphoneguard.permission.RECEIVE_BLOCK_BROADCAST">
    ...
</receiver>
```

### 1.4 无障碍服务被监听可能导致信息泄露

**位置**: `GuardAccessibilityService.kt`

**问题描述**:
无障碍服务监听了过多的事件类型，可能收集到用户敏感信息。

**代码**:
```kotlin
override fun onAccessibilityEvent(event: AccessibilityEvent) {
    when (event.eventType) {
        AccessibilityEvent.TYPE_WINDOW_STATE_CHANGED,
        AccessibilityEvent.TYPE_WINDOWS_CHANGED,
        AccessibilityEvent.TYPE_WINDOW_CONTENT_CHANGED,  // ⚠️ 可能泄露内容
        AccessibilityEvent.TYPE_ANNOUNCEMENT,            // ⚠️ 语音助手内容
        AccessibilityEvent.TYPE_ASSIST_READING_CONTEXT,   // ⚠️ 辅助阅读内容
        AccessibilityEvent.TYPE_GESTURE_DETECTION_START,
        AccessibilityEvent.TYPE_GESTURE_DETECTION_END -> {
            handleWindowEvent(event)
        }
    }
}
```

**风险**:
- `TYPE_WINDOW_CONTENT_CHANGED` 可能读取到屏幕内容
- `TYPE_ASSIST_READING_CONTEXT` 可能读取到用户意图

**建议**:
只监听必要的事件类型：
```kotlin
// 只监听窗口状态变化
AccessibilityEvent.TYPE_WINDOW_STATE_CHANGED,
AccessibilityEvent.TYPE_WINDOWS_CHANGED
```

**位置**: `AppKillerService.kt` 第 247-252 行，`GuardAccessibilityService.kt` 第 320-324 行

## 二、架构与设计问题

### 2.1 多个拦截服务重复且冲突

**问题描述**:
三个服务（GuardAccessibilityService、AppMonitorService、AppKillerService）功能重叠，导致：
- 重复检查和拦截
- 覆盖层闪烁
- 资源浪费
- 维护困难

**建议**:
统一为一个核心拦截服务，其他作为辅助：
```
核心拦截服务 (CoreBlockService)
├─> 事件监听（无障碍）
├─> 轮询检测（UsageStats）
├─> 统一决策引擎
└─> 统一执行层
```

### 2.2 全局锁双真源问题

**问题描述**:
全局锁状态存在于两个地方：
1. `SettingsManager.isGlobalLockEnabled()` - 全局设置
2. `AppRule.isGlobalLocked` - 应用级别

可能导致状态不一致。

**代码位置**:
```kotlin
// LockDecisionEngine.kt 第 39-45 行
val globalLocked = settingsManager.isGlobalLockEnabled()
val rule = appRuleRepository.getRuleByPackageName(packageName)

if (globalLocked || (rule?.isGlobalLocked == true)) {
    return BlockDecision(shouldBlock = true, reason = BlockReason.GLOBAL_LOCK, appName = appName)
}
```

**建议**:
统一为一个状态源，应用级别的全局锁应该直接引用全局设置。

### 2.3 OverlayService 静态状态管理不当

**位置**: `OverlayService.kt`

**问题描述**:
OverlayService 使用静态变量管理状态，多个服务并发修改，容易导致状态不一致。

**代码**:
```kotlin
companion object {
    private var overlayView: View? = null
    private var isShowing = false
    private var currentPackageName: String = ""
    // ...
}
```

**风险**:
- 多个服务同时调用 `showOverlay()` 和 `hideOverlay()`
- 竞态条件导致覆盖层闪烁或状态错乱

**建议**:
使用状态机或单例模式管理：
```kotlin
object OverlayController {
    private val state = AtomicReference<OverlayState>(OverlayState.Hidden)

    fun show(packageName: String, appName: String) {
        state.updateAndGet { current ->
            when (current) {
                OverlayState.Hidden -> {
                    actuallyShow(packageName, appName)
                    OverlayState.Shown(packageName)
                }
                else -> current
            }
        }
    }

    fun hide() {
        state.updateAndGet { current ->
            when (current) {
                is OverlayState.Shown -> {
                    actuallyHide()
                    OverlayState.Hidden
                }
                else -> current
            }
        }
    }
}

sealed class OverlayState {
    object Hidden : OverlayState()
    data class Shown(val packageName: String) : OverlayState()
}
```

### 2.4 规则引擎与执行层耦合

**问题描述**:
拦截决策和执行逻辑分散在多个服务中，难以维护和测试。

**当前流程**:
```
GuardAccessibilityService
  └─> LockDecisionEngine (决策)
       └─> GuardAccessibilityService.enforceBlock (执行)

AppKillerService
  └─> checkLimitConditions (自建决策)
       └─> forceStopApp (执行)

AppMonitorService
  └─> checkLimitConditions (自建决策)
       └─> forceShowOverlay (执行)
```

**建议**:
统一决策和执行层：
```kotlin
interface BlockExecutor {
    suspend fun executeBlock(packageName: String, reason: BlockReason)
    suspend fun executeUnblock(packageName: String)
}

class BlockCoordinator(
    private val decisionEngine: LockDecisionEngine,
    private val executor: BlockExecutor
) {
    suspend fun onAppSwitch(packageName: String) {
        val decision = decisionEngine.getBlockDecision(packageName)
        if (decision.shouldBlock) {
            executor.executeBlock(packageName, decision.reason)
        } else {
            executor.executeUnblock(packageName)
        }
    }
}
```

## 三、性能问题

### 3.1 过于频繁的轮询

**问题描述**:
多个服务使用高频率轮询，消耗 CPU 和电池。

| 服务 | 轮询间隔 | 频率 |
|-----|---------|------|
| AppKillerService | 50ms | 20次/秒 |
| AppMonitorService | 100ms | 10次/秒 |
| UsageTrackingManager | 3000ms | 0.33次/秒 |

**影响**:
- 高 CPU 使用率
- 电池快速消耗
- 可能被系统优化杀死

**建议**:
1. 减少轮询频率
   - AppKillerService: 50ms → 200-500ms
   - AppMonitorService: 100ms → 500-1000ms

2. 采用事件驱动 + 混合轮询策略：
```kotlin
// 主流程：事件驱动（无障碍）
GuardAccessibilityService.onAccessibilityEvent()

// 辅助流程：低频轮询（作为备份）
AppKillerService: 每 500ms 检查一次
```

### 3.2 重复的强制停止操作

**位置**: `GuardAccessibilityService.kt` 第 228-272 行

**问题描述**:
拦截时执行多次相同的强制停止操作，浪费资源。

**代码**:
```kotlin
handler.postDelayed({ tryForceStopApp(packageName) }, 120)
handler.postDelayed({ tryForceStopApp(packageName) }, 360)
handler.postDelayed({ tryForceStopApp(packageName) }, 700)
```

**建议**:
1. 先尝试一次，检测结果
2. 如果失败再重试，而不是盲目多次

```kotlin
private suspend fun forceStopAppWithRetry(packageName: String, maxRetries: Int = 3) {
    repeat(maxRetries) { attempt ->
        if (tryForceStopApp(packageName)) {
            return // 成功
        }
        delay(100 * (attempt + 1).toLong())
    }
    // 所有重试失败
    Log.w(TAG, "无法停止应用 $packageName")
}
```

### 3.3 数据库主线程访问风险

**问题描述**:
协程作用域配置可能不当，存在主线程访问数据库的风险。

**代码位置**:
```kotlin
// GuardAccessibilityService.kt
private val serviceScope = CoroutineScope(SupervisorJob() + Dispatchers.Main)

// AppKillerService.kt
private val serviceScope = CoroutineScope(SupervisorJob() + Dispatchers.Main)

// 数据库操作
serviceScope.launch {
    val rule = appRuleRepository.getRuleByPackageName(packageName)
    // ...
}
```

**风险**:
- `Dispatchers.Main` 中的协程可能阻塞主线程
- Room 数据库操作如果耗时，会导致 UI 卡顿

**建议**:
```kotlin
// IO 密集操作使用 Dispatchers.IO
serviceScope.launch(Dispatchers.IO) {
    val rule = appRuleRepository.getRuleByPackageName(packageName)
    // ...
}
```

### 3.4 内存泄漏风险

**位置**: 多个 Service 的 Handler 使用

**问题描述**:
Handler 持有 Service 的引用，如果 Handler 回调未及时清理，可能导致 Service 无法被回收。

**代码**:
```kotlin
private val handler = Handler(Looper.getMainLooper())

private val monitorRunnable = object : Runnable {
    override fun run() {
        checkAndKillApps()
        handler.postDelayed(this, MONITOR_INTERVAL)  // ⚠️ 持有 Service 引用
    }
}
```

**建议**:
使用 WeakReference 或在 onDestroy 中确保清理：

```kotlin
override fun onDestroy() {
    super.onDestroy()
    handler.removeCallbacksAndMessages(null)  // 清除所有回调
    serviceScope.cancel()
}
```

### 4.3 异常处理不当

**问题描述**:
多处空 catch 异常，只打印日志，可能隐藏真正的问题。

**代码示例**:
```kotlin
try {
    Runtime.getRuntime().exec("am force-stop $packageName")
} catch (e: Exception) {
    // ⚠️ 忽略，可能没有root权限
}
```

**建议**:
1. 区分预期异常和非预期异常
2. 对于预期异常，记录并降级处理
3. 对于非预期异常，应该向上抛出或告警

```kotlin
try {
    Runtime.getRuntime().exec("am force-stop $packageName")
} catch (e: SecurityException) {
    // 预期：没有权限
    Log.w(TAG, "Shell命令需要root权限", e)
} catch (e: IOException) {
    // 预期：命令执行失败
    Log.w(TAG, "Shell命令执行失败", e)
} catch (e: Exception) {
    // 非预期：记录并上报
    Log.e(TAG, "Shell命令执行异常", e)
    reportError(e)
}
```

### 9.2 数据迁移策略缺失

**位置**: `AppDatabase.kt`

**问题描述**:
数据库使用 `fallbackToDestructiveMigration()`，版本升级时会丢失数据。

**代码**:
```kotlin
Room.databaseBuilder(...)
    .fallbackToDestructiveMigration()  // ⚠️ 升级时清空数据
    .build()
```

**建议**:
实现数据迁移：
```kotlin
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(database: SupportSQLiteDatabase) {
        // 迁移逻辑
    }
}

Room.databaseBuilder(...)
    .addMigrations(MIGRATION_1_2)
    .build()
```
