# 拦截机制分析与问题报告

## 文档说明
本文档分析了 KidsPhoneGuard 应用中多种拦截机制的实现情况，以及发现的关键问题。

**创建日期**: 2026-03-15
**分析范围**: 拦截服务架构、时长限制实现、冲突分析

---

## 一、当前拦截服务架构

### 1.1 活跃的拦截服务

应用目前有 **4 个拦截相关的服务**，其中 3 个活跃运行：

| 服务名称 | 状态 | 监控频率 | 主要功能 |
|---------|------|---------|---------|
| GuardAccessibilityService | ✅ 活跃 | 事件驱动（500ms 防抖） | 无障碍服务监听窗口变化 |
| AppMonitorService | ✅ 活跃 | 每 100ms 轮询 | 持续监控顶层应用 |
| AppKillerService | ✅ 活跃 | 每 50ms 轮询 | 强制停止应用进程 |
| AppBlockerService | ❌ 已禁用 | - | 代码中已禁用，不执行任何操作 |

### 1.2 服务详细分析

#### GuardAccessibilityService（无障碍服务）

**文件位置**: `app/src/main/java/com/kidsphoneguard/service/GuardAccessibilityService.kt`

**工作原理**:
- 监听无障碍事件（窗口状态变化等）
- 使用 500ms 防抖间隔避免重复处理
- 调用 `LockDecisionEngine` 判断是否需要拦截

**拦截动作序列**（当检测到需要拦截时）:
```kotlin
// 时间线
0ms    -> 显示覆盖层
0ms    -> performGlobalAction(GLOBAL_ACTION_BACK)
120ms  -> tryForceStopApp() (第1次)
360ms  -> tryForceStopApp() (第2次)
700ms  -> tryForceStopApp() (第3次)
820ms  -> performGlobalAction(GLOBAL_ACTION_HOME)
```

`tryForceStopApp()` 包含 4 种强制停止方法:
1. `task.finishAndRemoveTask()` - 结束任务并移除
2. `Process.killProcess()` - 杀死进程（Android 8.0 以下）
3. `activityManager.killBackgroundProcesses()` - 杀后台进程
4. 反射调用 `forceStopPackage()` - 系统级强制停止
5. `Runtime.exec("am force-stop")` - Shell 命令强制停止

**冷却机制**:
- 同一应用在 5 秒内不会重复拦截
- 拦截后有 700ms 的"保持期"，期间不再处理新事件

---

#### AppMonitorService（应用监控服务）

**文件位置**: `app/src/main/java/com/kidsphoneguard/service/AppMonitorService.kt`

**工作原理**:
- 每 100ms 轮询一次顶层应用
- 使用 `UsageStatsManager` 获取最近使用的应用

**拦截动作**:
- **只显示覆盖层**，不主动杀死应用
- 重复显示覆盖层 4 次（间隔 0ms, 50ms, 150ms, 300ms）

**时段限制检查**:
- 检查全局锁状态
- 检查禁用时段（`blockedTimeWindows`）
- **不检查时间限制**

---

#### AppKillerService（应用强制停止服务）

**文件位置**: `app/src/main/java/com/kidsphoneguard/service/AppKillerService.kt`

**工作原理**:
- 每 50ms 轮询一次（频率最高）
- 同时检查 `UsageEvents` 和顶层应用

**拦截动作**（7 种方式）:
1. `task.finishAndRemoveTask()` - 结束任务
2. `Process.killProcess()` - 杀进程（Android 8.0 以下）
3. `activityManager.killBackgroundProcesses()` - 杀后台进程
4. `Runtime.exec("am force-stop")` - Shell 命令
5. 反射调用 `forceStopPackage()` - 系统级强制停止
6. 启动桌面（`startActivity` with HOME intent）
7. 发送自定义广播 `ACTION_KILL_APP`

**冷却机制**:
- 同一应用在 500ms 内不会重复杀死

**时段限制检查**:
- 检查全局锁状态
- 检查禁用时段（`blockedTimeWindows`）
- **不检查时间限制**（关键问题！）

## 二、冲突分析：为什么会出现"拉扯"现象

### 2.1 重复拦截现象

由于多个服务同时监控同一应用，导致：

| 场景 | GuardAccessibilityService | AppMonitorService | AppKillerService | 总计 |
|-----|----------------------|------------------|------------------|-----|
| 应用被打开 | 1 次 BACK + 3 次停止 | 4 次显示覆盖层 | 7 种停止方法 | 大量重复操作 |
| 覆盖层显示 | 有 | 重复显示 4 次 | 重复显示 | 覆盖层闪烁 |

### 2.2 决定性拦截预测

基于代码分析，**AppKillerService 最可能起到决定性作用**，原因：

1. **轮询频率最高**: 50ms vs 100ms vs 500ms
2. **拦截方式最多**: 7 种不同的强制停止方法
3. **不依赖无障碍服务**: 直接通过系统 API 操作

**但是**，AppKillerService 的 `checkLimitConditions()` 函数存在缺陷：

```kotlin
// AppKillerService.kt 第 295-307 行
private suspend fun checkLimitConditions(rule: AppRule): Boolean {
    if (rule.isGlobalLocked) {
        return true
    }

    if (rule.blockedTimeWindows.isNotEmpty()) {
        if (isInBlockedTimeWindow(rule.blockedTimeWindows)) {
            return true
        }
    }

    return false  // ⚠️ 这里没有检查时间限制！
}
```

这个函数**只检查全局锁和禁用时段，完全忽略时间限制**。

### 2.3 拦截链路图

```
应用打开
   │
   ├─> GuardAccessibilityService (事件驱动)
   │    └─> LockDecisionEngine → 拦截或放行
   │         ├─> BACK
   │         ├─> 3次强制停止
   │         └─> HOME
   │
   ├─> AppMonitorService (100ms 轮询)
   │    └─> 检查时段限制 → 显示覆盖层 x4
   │
   └─> AppKillerService (50ms 轮询)
        └─> 检查时段限制 → 7种方式强制停止
```

## 三、关键问题：时间限制实现缺陷

### 3.1 问题概述

**核心问题**：当应用的时间限制到达时，**只有覆盖层会显示，应用不会被主动关闭**。

### 3.2 根因分析

#### 检查时长限制的组件

| 组件 | 检查时长限制 | 触发方式 | 拦截能力 |
|-----|------------|---------|---------|
| LockDecisionEngine | ✅ 是 | 被其他服务调用 | 返回决策，不直接拦截 |
| UsageTrackingManager | ✅ 是 | 每 3 秒轮询 | 只发送广播/显示覆盖层 |
| AppMonitorService | ❌ 否 | - | - |
| AppKillerService | ❌ 否 | - | - |

#### 时长限制的实际工作流程

```
1. UsageTrackingManager 每 3 秒检查一次
   ├─> 统计使用时长
   ├─> 更新数据库
   └─> 检查是否超限

2. 如果超限：
   ├─> 检查 GuardAccessibilityService 是否运行
   │    ├─> 是 → 发送 ACTION_BLOCK_APP 广播
   │    └─> 否 → 直接显示覆盖层
   └─> ⚠️ 没有强制关闭应用的逻辑！

3. GuardAccessibilityService 收到广播后：
   └─> 执行拦截动作（BACK + 强制停止 + HOME）
```

**问题所在**：

1. **如果无障碍服务被暂停/被杀死**（MIUI 等系统的常见情况）：
   - UsageTrackingManager 只会显示覆盖层
   - 用户可以关闭覆盖层后继续使用应用
   - 应用**不会被主动关闭**

2. **即使无障碍服务正常运行**：
   - 需要等待最多 3 秒才能检测到超限
   - 这期间用户可能已经进入应用深处
   - 用户体验不佳

### 3.3 代码证据

**UsageTrackingManager.kt 第 133-153 行**:
```kotlin
private suspend fun checkAndTriggerBlock(app: KidsPhoneGuardApp, packageName: String) {
    val rule = app.appRuleRepository.getRuleByPackageName(packageName) ?: return

    // 只检查LIMIT类型的应用
    if (rule.ruleType != RuleType.LIMIT) return

    // 检查时长限制
    if (rule.dailyAllowedMinutes > 0) {
        val todayUsage = app.dailyUsageRepository.getTodayUsageSeconds(packageName)
        val allowedSeconds = rule.dailyAllowedMinutes * 60L

        if (todayUsage >= allowedSeconds) {
            // 触发拦截，显示覆盖层
            if (GuardAccessibilityService.isServiceRunning()) {
                BroadcastPermissionHelper.sendBlockAppBroadcast(app, packageName)
            } else {
                OverlayService.showOverlay(app, packageName, rule.appName)
            }
            // ⚠️ 这里只有显示覆盖层，没有强制关闭！
        }
    }
}
```

## 四、白名单安全问题

### 4.1 问题描述

`WhitelistManager.kt` 使用 `contains()` 进行包名匹配，存在安全风险：

```kotlin
fun isInWhitelist(packageName: String): Boolean {
    return SYSTEM_WHITELIST.any { packageName.contains(it, ignoreCase = true) }
}
```

### 4.2 安全风险

这种匹配方式存在**包名伪装绕过风险**：

| 白名单项 | 恶意应用包名 | 结果 |
|---------|--------------|------|
| `com.miui.home` | `com.miui.home.evil.app` | ✅ 被放过（危险！）|
| `com.android.settings` | `evil.com.android.settings.fake` | ✅ 被放过（危险！）|

### 4.3 建议修复

使用精确匹配或严格控制前缀：

```kotlin
fun isInWhitelist(packageName: String): Boolean {
    // 方案1: 精确匹配
    return SYSTEM_WHITELIST.contains(packageName)

    // 方案2: 严格前缀匹配（仅针对系统包）
    return SYSTEM_WHITELIST.any {
        packageName == it || packageName.startsWith("$it.")
    }
}
```

## 五、其他发现

### 5.1 广播安全问题

`BroadcastPermissionHelper.kt` 使用未受保护的自定义广播：

```kotlin
const val ACTION_BLOCK_APP = "com.kidsphoneguard.action.BLOCK_APP"
const val ACTION_KILL_APP = "com.kidsphoneguard.action.KILL_APP"
```

**风险**: 外部应用可以发送这些广播来触发拦截，干扰正常功能。

**建议**: 使用 `RECEIVER_NOT_EXPORTED` 或添加权限保护。

### 5.2 覆盖层竞争

多个服务都在调用 `OverlayService.showOverlay()`：
- GuardAccessibilityService
- AppMonitorService
- AppKillerService
- UsageTrackingManager

这可能导致覆盖层闪烁或显示/隐藏状态不一致。

### 5.3 重复的代码

`AppMonitorService` 和 `AppKillerService` 都有相似的 `checkLimitConditions()` 函数，都缺少时间限制检查。

## 六、建议改进

### P0（关键修复）

1. **修复时间限制拦截问题**
   - 在 `AppKillerService` 和 `AppMonitorService` 的 `checkLimitConditions()` 中添加时间检查
   - 或者在 UsageTrackingManager 超限时直接调用强制停止

2. **修复白名单安全风险**
   - 改用精确匹配或严格前缀匹配

3. **统一拦截机制**
   - 保留一个主要的拦截服务
   - 其他服务作为辅助，避免重复操作

### P1（重要改进）

1. **修复广播安全问题**
   - 使用保护性广播或权限验证

2. **解决覆盖层竞争**
   - 统一覆盖层控制逻辑
   - 添加状态锁

### P2（性能优化）

1. **减少轮询频率**
   - AppKillerService 从 50ms 调整到 200ms
   - AppMonitorService 从 100ms 调整到 500ms

2. **优化重复操作**
   - 合并多个服务的拦截动作
   - 减少不必要的重复调用
